# コーディング規約

このプロジェクト（Ruby on Rails 7.1 / Ruby 3.0.6）でコードを作成・編集する際は、必ず以下の規約に従うこと。

## 基本方針

- コードは常にこの規約に則って作成する
- 規約に反するコードは、たとえ動作しても修正する

## Ruby 命名規則

- 変数・メソッド名: `snake_case`
- クラス・モジュール名: `CamelCase`
- 定数: `SCREAMING_SNAKE_CASE`
- ファイル名: `snake_case.rb`
- 真偽値を返すメソッドには `?` サフィックスを付ける（例: `valid?`, `admin?`）
- 破壊的メソッドには `!` サフィックスを付ける（例: `save!`, `update!`）

## Rails 規約

- MVC パターンを厳守する（ビジネスロジックはモデルまたはサービスオブジェクトに置く）
- ルーティングは RESTful に設計する（`resources` を優先して使う）
- コントローラはスリムに保つ（1 アクション 10 行以内を目安）
- N+1 クエリを避ける（`includes`, `eager_load`, `preload` を適切に使う）
- バリデーションはモデルに定義する
- スコープは `scope` メソッドで定義する

## コードスタイル

- インデント: スペース 2 つ（タブ禁止）
- 1 行の最大文字数: 120 文字
- メソッドの末尾に空行を入れない
- `do...end` と `{...}` の使い分け: 複数行は `do...end`、単行は `{...}`
- 文字列リテラルはシングルクォート `'` を使う（式展開が必要な場合はダブルクォート `"`）
- `unless` は否定条件が 1 つの場合のみ使う（`else` と組み合わせない）
- 三項演算子は単純な条件のみ（ネスト禁止）

## モデル

- アソシエーションを最初に定義する
- バリデーションをアソシエーションの次に定義する
- スコープをバリデーションの次に定義する
- コールバックはスコープの後に定義する
- クラスメソッド → インスタンスメソッドの順に定義する

## テスト（RSpec）

- `describe` にはクラス名またはメソッド名を指定する
- `context` は条件を表す文（「〜の場合」）で記述する
- `it` は期待する振る舞いを 1 つだけ記述する
- `let` / `let!` を使ってデータのセットアップを行う
- `before` はセットアップ処理のみに使い、アサーションを含めない
- FactoryBot でテストデータを作成する

## セキュリティ

- ユーザー入力は必ずバリデーションまたはサニタイズを行う
- SQL インジェクション防止のため、プレースホルダ（`?` または名前付きバインド）を使う
- Strong Parameters を必ずコントローラに実装する
- パスワードは `bcrypt` でハッシュ化する（平文保存禁止）
- 認証・認可のチェックを各コントローラアクションに実装する

## その他

- マジックナンバーは定数として定義する
- コメントは「なぜ（WHY）」を説明する場合のみ書く（何をしているかはコードで表現する）
- 不要なコード（コメントアウトされたコード、デバッグ用 `puts` など）は残さない
